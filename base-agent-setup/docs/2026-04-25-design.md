# base-agent-setup — Design Document

**Date:** 2026-04-25
**Author:** Leo + Claude (brainstorming session)
**Status:** Approved for writing-plans
**Slogan:** 30-minute base-agent setup, done right, every time.

---

## Context

This skill is the workhorse of **VoiceAIMachine GTM Phase 1** — automating SpotFunnel's own manual customer-onboarding flow (the one we do today by hand for Teleca, TelcoWorks, Trentham) down to roughly 30 minutes of Leo's attention at the same or higher quality than today's hand-built agents.

Leo meets with a new customer, records the meeting (in-built whisper transcription produces a text transcript), invokes `/base-agent` from any project directory on his laptop, and walks away with:

1. A rough voice-AI agent live in Ultravox, knowledge-based on the customer's business, running with Teleca/TelcoWorks-proven voice/temperature/inactivity settings, no tools or call flows yet.
2. A real Telnyx DID from the pool, TeXML-wired at 16 kHz, pointed at the rough agent.
3. The agent fully registered in the SpotFunnel dashboard with call-summary emails routed to `leo@getspotfunnel.com` (transferred to the customer later, manually, before go-live).
4. A massive ChatGPT-ready **discovery prompt** the customer pastes into a fresh ChatGPT conversation, which interviews them using their website + meeting as known context and produces a detailed brief Leo receives back by email with zero refinement needed from the customer.

After the brief returns, tools and call flows are designed and added in a separate, future step (not in scope for this skill).

## Goals

1. **Single invocation, end-to-end.** One `/base-agent [name]` command produces everything above.
2. **Never miss a step.** All 11 stages run in a checklist-enforced order with a resumable state file.
3. **Minimize redundant questions.** Skill infers from website + meeting + reference settings; asks the operator only for genuinely novel info.
4. **Quality parity with hand-built agents.** Voice, temperature, inactivity messages, first-speaker config copied live from a configured reference agent — so any tuning to that agent propagates to every new customer automatically.
5. **Opus 4.7 is the brain.** Every inference step (brain-doc synthesis, prompt authoring, discovery-prompt generation) runs inline in the Claude Code conversation. No Anthropic API / Haiku calls — the operator's Max subscription is doing the work.
6. **Globally invokable.** Skill folder symlinked or copied into `~/.claude/skills/base-agent-setup/` — runs from any project directory.
7. **Portable across machines.** The skill, its sibling `onboard-customer` skill, and all context they pull (universal rules, reference docs, runbooks, schema migrations) live in a single public GitHub repo. A new operator clones the repo, symlinks the skill folders into their `~/.claude/skills/`, fills in `.env` from `.env.example`, and runs the full pipeline against their own accounts (Ultravox, Telnyx, Supabase, Resend, Firecrawl) without needing access to the original author's filesystem. Zero hardcoded local paths or account-specific identifiers in committed files.

## Non-goals

- Tool design (CRM integration, calendar booking, SMS, transfer wiring) — that's the post-brief step, separate skill.
- Stress testing — existing `voice-stress-test` skill handles that after tools are added.
- Customer email transfer (Leo → customer) — manual step before go-live.
- Telnyx number purchasing — Leo buys numbers manually in bulk; skill only *claims* from the existing pool and alerts if empty.
- Supporting white-label partners (Teleca, TelcoWorks rebranding) — this skill is Leo-only internal tooling for GTM Phase 1. GTM Phase 2 productization is a future project.
- Translating the skill to a productized API-backed service — GTM Phase 3, years away.

## The flow — 11 stages

Every stage writes to `runs/{slug}-{timestamp}/state.json` on completion. Re-invocation with the same slug resumes from the last successful stage.

### Stage 0 — Env preflight (portable)

Resolve env file location in this order:
1. `$SPOTFUNNEL_SKILLS_ENV` if set
2. `<repo-root>/.env` (works when the operator clones the repo and follows `INSTALL.md`)
3. Fall back: prompt the operator once for their env file path; cache that path at `~/.config/spotfunnel-skills/env-path` for next runs

Required vars (each documented in `.env.example` at repo root with a comment explaining where to get the value):
- `ULTRAVOX_API_KEY`, `TELNYX_API_KEY`, `FIRECRAWL_API_KEY`
- `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`
- `RESEND_API_KEY`, `RESEND_FROM_EMAIL`, `OPS_ALERT_EMAIL`
- `REFERENCE_ULTRAVOX_AGENT_ID` — operator's chosen reference agent
- `TELNYX_POOL_TEXML_APP_ID` — operator's TeXML app
- `DASHBOARD_SERVER_URL` — operator's dashboard-server URL (used by `/onboard-customer`)
- `N8N_BASE_URL`, `N8N_API_KEY`, `N8N_ERROR_REPORTER_WORKFLOW_ID` — for `/onboard-customer` Stage 7b

Halt with a clear "missing X — see `.env.example` for what to put here" message if anything required is empty.

### Stage 1 — Gather inputs

Ask the essential questions above, one at a time. Skip any already answered in the invocation args or from recent conversation context. Infer slug from customer name, show Leo for confirmation.

### Stage 2 — Firecrawl scrape (async)

Kick off Firecrawl full-site crawl with `respectRobots: false`. Dump raw markdown + sitemap to `runs/{slug}-*/scrape/`. Runs concurrent with Stage 5.

### Stage 3 — Brain-doc synthesis (Claude inline)

Claude Opus 4.7 reads the full scrape + meeting transcript + operator hints, produces a structured brain-doc:
- Company identity (name, trading name, one-line pitch)
- Services / products with descriptions
- Hours (from site, verified/corrected from meeting)
- Staff names and roles mentioned
- Locations / service area
- Existing phone + email contact
- Any explicit policies, prices, guarantees mentioned
- Tone markers (formal/casual, warmth, humor)

Written to `runs/{slug}-*/brain-doc.md`. Size target: 3–8 KB. Flagged sections mark "inferred from meeting, not site" vs. "confirmed by both."

### Stage 4 — Rough system prompt generation (Claude inline)

Claude composes the agent's system prompt:
```
UNIVERSAL_RULES    (the 16-rule base, lifted from VoiceAIMachine composer)
+ AGENT_IDENTITY   ("You are [name] for [company]. ...")
+ BRAIN_DOC        (from Stage 3)
+ MINIMAL_TOOL_NOTE ("You have no action tools today. For any request requiring a tool, say so and offer to take a message or warm-transfer.")
```

Written to `runs/{slug}-*/system-prompt.md`. Deliberately **no** call flows, **no** integration references, **no** tool schemas. The agent can converse intelligently about the business and gracefully degrade when asked to do something.

### Stage 5 — Reference agent settings pull

`GET https://api.ultravox.ai/api/agents/{REFERENCE_AGENT_ID}` (TelcoWorks Jack by default). Extract:
- `voice` (ElevenLabs voice ID + weight)
- `temperature`
- `firstSpeaker` / first-speaker settings
- `inactivityMessages` (the full tiered sequence)
- `languageHint`, `transcriptOptional`, any other speech-side config
- Model selection

Cached to `runs/{slug}-*/reference-settings.json` for audit. **Live pull every run** — so tuning propagates automatically.

### Stage 6 — Create Ultravox agent

`POST https://api.ultravox.ai/api/agents` with:
- `name`: `{Customer}-{AgentFirstName}` (e.g. `AcmePlumbing-Steve`)
- `systemPrompt`: output of Stage 4
- All settings from Stage 5
- **No** `selectedTools`
- `callTemplate.eventMessages[]`: empty for now — the call-ended webhook gets set manually by Leo in the Ultravox console (per CLAUDE.md rule: never PATCH agents via API). The `/onboard-customer` skill's Stage 7 message already reminds Leo of this one-click step.

Capture `agent_id` in state. **Never PATCH this agent** — any correction means POSTing a new agent and discarding the old.

### Stage 7 — Claim Telnyx DID from pool

Query Telnyx for unassigned DIDs in the pool matching preferred area code (inferred from scraped address; asked if ambiguous; falls back to any Australian DID if preferred unavailable).

**If pool has < 3 DIDs remaining:** skill still claims one but fires a Resend alert to `leo@getspotfunnel.com` with subject `[VoiceAIMachine] DID pool low — {N} remaining`.

**If pool is empty:** skill halts, fires a hard Resend alert, prints `Pool exhausted — buy more DIDs and re-run /base-agent {slug} to resume`. Never auto-buys.

### Stage 8 — TeXML app wiring

Reuse the existing `VoiceAIMachine-Hot` TeXML app (per `phase-4-handoff-2026-04-23.md`): its `status_callback` is already pointed at VAM's `/telnyx-webhook`, and the `claim_did_for_user` path pattern from VAM's `services/api/src/api/telnyx.py` is the authoritative reference. Skill mirrors that logic:
- Verify the DID is bound to the pool TeXML app
- Confirm codec is 16 kHz (g711 fallback OK)
- Confirm `status_callback` still points at the VAM webhook

### Stage 9 — TeXML → Ultravox telephony_xml wiring

Update the TeXML app to point the claimed DID at:
`https://app.ultravox.ai/api/agents/{agent_id}/telephony_xml`

Verify by dialing the DID's TeXML endpoint and checking the returned XML references the correct agent_id. (Low-cost — no real call placed at this stage; cURL check only.)

### Stage 10 — Per-customer discovery prompt (Claude inline)

**Output sizing for ChatGPT paste limits:** ChatGPT free-tier per-message paste reliably handles ~30k characters; long meeting transcripts can push the full prompt over that. Skill auto-routes:

- **Combined size ≤ 25k chars:** emit ONE `discovery-prompt.md` containing methodology + opener + brain-doc + transcript. Customer pastes once.
- **Combined size > 25k chars:** emit TWO files — `discovery-prompt.md` (methodology + opener + scope statement, always under 10k chars) and `customer-context.md` (brain-doc + full meeting transcript). Customer pastes the prompt as their first message and drag-drops the context file as an attachment. Cover email explains both paths so Leo doesn't need to think about it per-customer.


Claude Opus 4.7 writes a bespoke discovery prompt using:
- The scrape summary (from Stage 3 brain-doc)
- The full meeting transcript
- Leo's operator hints
- The reference doc at `reference-docs/discovery-methodology.md` (methodology, coverage targets, posture, output schema)

Output: a single copy-pasteable text block (6–15k words typical) that the customer pastes into a fresh ChatGPT conversation. The prompt:
- States known facts about the business as ground truth (don't re-ask)
- Embeds the transcript with cited references ("from your meeting: ...")
- Opens with a tailored first question anchored in something specific the customer said
- Instructs ChatGPT to follow the methodology: open-ended, brain-engaging, web-research on compliance, options-not-questions on anything regulatory, graceful on hard asks
- Specifies the brief output schema (6 sections A–F) as copy-pasteable prose

Also generates a **short cover email template** Leo forwards to the customer, with 1 paragraph of context + the "paste this into ChatGPT" instruction + the prompt appended.

Written to `runs/{slug}-*/discovery-prompt.md` and `runs/{slug}-*/cover-email.md`.

### Stage 11 — Hand off to `/onboard-customer`

Invoke via the Skill tool with pre-filled inputs:
- `slug`
- `name`
- `ultravox_agent_ids: [agent_id]`
- `telnyx_numbers: [claimed_did]`
- `primary_user_email: leo@getspotfunnel.com`
- `primary_user_name: Leo Gewert`
- `archetype: auto` (onboard-customer infers — new vertical OR Teleca/TelcoWorks clone)

The existing `onboard-customer` skill handles: workspace insert, auth.users invite, public.users insert, n8n error reporting wiring, magic-link generation, verification checklist.

On failure of `onboard-customer`, the skill surfaces the error but leaves the Ultravox agent + Telnyx DID claimed — Leo can run `/onboard-customer` manually with the state file's saved values.

### Coverage targets (the brief must answer — *within the scope defined by the meeting*)

> Every coverage target below is conditional on being inside the scope the customer defined in the meeting. If the scope is narrow (e.g. "appointment setter only"), entire sections below may be out of scope — don't ask them. If the scope is broad ("full inbound receptionist"), all sections apply.

**A. Callers & dreams**
- Caller personas (confirmed/corrected from site; *not* re-asked — ChatGPT asserts what it inferred, customer edits)
- Dream call ending per persona (open question, brain-engaging)

**B. Transfers & humans**
- Transfer targets: name, role, phone number, which calls route to whom
- After-hours + escalation policy:
  - Is there an emergency number for after hours, OR should the agent urge the caller to leave a message / be called back next business day?
  - If an emergency number exists: what scenarios permit it? (Protects the person on that number from 3am "what are your prices" calls.)

**C. Integrations & dream outputs**
- CRM / calendar / email / SMS / accounting / backend in use (with versions where relevant). For each named tool, ChatGPT does a web search to verify API access tier (self-serve vs long-approval vs none) per principle #6 above before responding.
- Per-integration dream behavior (specific, e.g. "create contact in Salesforce stage 'Inbound Lead' with call summary in notes field"). Default response is *yes we can do that*; only flag if research surfaces a genuine API blocker.
- Wishlist items genuinely beyond reach (custom internal systems with no public API only) — flagged for team review with the natural fallback offered alongside.

**D. Voice, brand, red lines**
- Brand voice (formal/casual, warmth level, signature phrases, do-not-say words)
- Red-line topics (pricing quotes, legal advice, competitor comparison, internal staff names, anything else)

**E. Compliance & edge cases** (ChatGPT does web research → offers options)
- Industry compliance (medical, legal, financial, debt-collection, trade-licensed, alcohol, etc.)
- **Call recording — do they want calls recorded?** With jurisdiction disclosure rules baked into the same question. ChatGPT researches state/country requirements, presents them alongside the yes/no ask.
- Known failure modes today (what goes wrong currently, what causes complaints)

**F. Operational context**
- Business hours, peak times, volume estimate
- Agent operating hours (if not already in meeting): 24/7 vs. business-hours only vs. after-hours only vs. weekends only

### Question-generation principles

1. **Meeting-first scope inference (READ THIS FIRST).** The meeting transcript is the PRIMARY source for the customer's intent. Read it before anything else and identify the *scope* of what the customer wants this agent to do — is it a full inbound receptionist with transfers? An appointment setter only? A lead qualifier for ad traffic? A customer-support line for one product? **The website is secondary context for facts; it does not expand scope.** If the customer says in the meeting "I only want an inbound appointment setter for Google Ads leads," do NOT ask about transfer targets or walk-in customers or product returns — those are outside the scope the customer defined. When ambiguous, confirm scope explicitly up front: *"Based on our call, I'm understanding this agent's job is narrowly [X] — should I also cover [Y] or is that out of scope?"*
2. **Never re-ask anything answered in the site or meeting.** Assert it as known, invite correction. *"Based on your site and our call, your busiest time looks like Monday mornings 8–11am — is that right?"*
3. **Cite the source when following up.** *"You mentioned in our call that when new leads call, you want them triaged before reaching your sales team. Let's dig into that..."*
4. **Compliance / regulatory / technically-ambiguous questions get research first.** ChatGPT does a web search, returns with a short summary + two viable options (A/B), asks which fits. Pattern: *"You said you want debt-collection calls handled by the AI. I've pulled up the current Australian rules under the ACL and National Credit Code — these two patterns would both comply, which matches how you operate? Option A: [...] Option B: [...]"*
5. **Every question is open and brain-engaging.** The customer does 80% of the writing. Skill pushes them to articulate dreams and specifics, not tick boxes.
6. **Integration optimism (LOAD-BEARING — read carefully).** Assume the build team can integrate with **any tool that has a publicly-accessible API that doesn't require a multi-week developer-program approval.** That's the ceiling. When the customer names a tool, ChatGPT must research it before responding:
   - **Self-serve standard API** (HubSpot, Salesforce, Google Workspace / Calendar / Gmail, Microsoft 365 / Outlook, Calendly, Pipedrive, Zoho, Notion, Airtable, Slack, Discord, ClickUp, Monday, Asana, Stripe, Xero, MYOB, ServiceM8, Jobber, Tradify, simPRO, Cliniko, Halaxy, MINDBODY, etc.) → respond confidently *"Yes, we can do that"*. No caveats. No "we'll see if it's possible." Just yes.
   - **Long-approval API** (some banking APIs, certain telephony aggregators, Meta WhatsApp Business API in some regions, Apple PassKit, healthcare networks like Medicare/Bupa direct integrations) → flag honestly: *"This one typically needs a 2–3 week developer-program approval, so it adds lead time. The team will assess."*
   - **No public API at all** (custom in-house backend with no exposed endpoints, legacy on-prem systems, niche bespoke software) → THIS is the genuine "may not be feasible" case — flag for the team and offer the natural fallback (email summary, SMS message, etc.).
   - **Default to YES.** The vast majority of tools SMBs use today have self-serve APIs. Being overly conservative here is the most expensive failure mode — it costs us deals.
7. **Hard asks handled gracefully** *(only when genuinely hard per principle 6).* *"Your current backend is custom-built with no public API, so directly-into-inventory writes may not be feasible without extra work. I'll flag it for the team. In the meantime, the natural fallback is an email summary you action manually — would that bridge the gap?"*

### Posture rules

- Optimistic but realistic
- Stay focused on the customer's business outcomes — what should happen on a call, what integrations they want, what their callers need
- Output as long as needed — multiple turns if the brief is big, no holding back
- Final brief is copy-pasteable prose, no rich formatting, ready to email back with zero refinement

### Brief output schema

Six sections mirroring coverage A–F, prose with embedded lists. Single block, customer pastes into email back to Leo.

## Claimed idempotency & failure modes

| Failure | Behavior |
|---|---|
| Env var missing at Stage 0 | Halt with clear "paste X" message |

| Reference agent GET fails | Halt; reference agent is load-bearing |

| `onboard-customer` failure | Surface error, leave agent + DID claimed, save state — Leo re-runs manually |
| Discovery prompt too long | Claude chunks into multi-turn (ChatGPT-side) — notes in cover email |

## Success criteria

1. A new customer goes from "meeting done" to "rough agent + discovery prompt ready to send" in under 30 minutes of Leo's attention.
2. The rough agent can hold a coherent conversation about the customer's business on first try — tested by dialing it.
3. The discovery prompt, pasted into ChatGPT, produces a brief that covers sections A–F with enough specificity that Leo can design tools from it without further Q&A with the customer.
4. Call recording jurisdiction rules are surfaced as options (not generic advice) in every brief where recording is requested.
5. Zero cases of the skill silently skipping a stage or producing a broken Ultravox agent.

## Future consideration — Custom GPT delivery (Phase 2+)

For Phase 1, the discovery prompt is delivered as a copy-paste block the customer drops into their own ChatGPT, and the brief comes back via email. This works but requires a final copy-paste from customer → Leo.

A future upgrade once Phase 1 has 5+ customers/month: build a single shared Custom GPT (e.g. "Spotfunnel Voice Brief Builder") that holds the methodology in knowledge files and exposes a `submitBrief` Action pointing at a SpotFunnel API endpoint. Per-customer, the skill generates a much shorter bespoke prompt (scrape summary + transcript + tailored opener) that the customer pastes as message 1; the GPT auto-POSTs the final brief structured-JSON to us at the end of the conversation, eliminating manual copy-paste.

**Blockers right now:** customer must have ChatGPT Plus/Team/Enterprise to use Custom GPTs (real friction wall for SMB on free tier); Action maintenance + auth rotation overhead; methodology is iterating weekly so re-shipping the GPT each time is heavier than re-generating the prompt. Revisit once methodology is stable and volume justifies the build.

OpenAI confirms creators do NOT see user conversations — Actions are the only data egress. Source: [GPTs Data Privacy FAQ](https://help.openai.com/en/articles/8554402-gpts-data-privacy-faq).

## Open questions (resolve before or during implementation)

- **Reference agent ID.** Each operator picks their own. Leo's default is TelcoWorks-Jack — the actual agent_id goes in `.env` only, never committed.

- **Agent name pattern.** `{Customer}-{AgentFirstName}` (matches existing convention).
- **Resend from address.** Configurable via `RESEND_FROM_EMAIL`; `.env.example` suggests `noreply@<your-domain>`.
